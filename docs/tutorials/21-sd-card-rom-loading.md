# Tutorial 21 — SD Card ROM Loading

Up to now our Game Boy has been running tiny test programs embedded directly in
FPGA distributed RAM via `$readmemh`. That's great for testing but useless for
real games. The Tang Nano 20K has a built-in microSD card slot — this tutorial
adds an SPI SD card controller, a minimal FAT32 parser, and an auto-boot loader
that finds the first `.gb` file on the card and loads it into 32 KB of BSRAM.
When boot completes, the CPU comes out of reset and the game runs.

SDRAM support (for games larger than 32 KB ROM) is deferred to Tutorial 22. For
now, BSRAM limits us to ROM-only games up to 32 KB — Tetris, Dr. Mario,
Alleyway, and many homebrew titles work perfectly.

## Architecture Overview

The boot sequence runs automatically at power-on:

```
Power-on → SD Card Init → Read MBR → Read VBR → Scan Root Dir
  → Find first .gb → Load ROM into BSRAM → Release CPU reset → Game runs
```

Three new modules form a layered stack:

```
sd_spi     ← SPI byte transport (SCLK/MOSI/MISO/CS)
  ↑
sd_reader  ← Card init (CMD0→CMD8→ACMD41→CMD16) + sector read (CMD17)
  ↑
sd_boot    ← FAT32 parse (MBR→VBR→root dir) + ROM loader
  ↑
gb_top     ← System state, BSRAM ROM, address mux
```

## SD Card Pins

The Tang Nano 20K's built-in microSD slot uses these FPGA pins in SPI mode:

| Signal | SPI Function | FPGA Pin |
|--------|-------------|----------|
| sd_clk | SCLK | 83 |
| sd_cmd | MOSI | 82 |
| sd_dat0 | MISO | 84 |
| sd_dat3 | CS (active low) | 81 |
| sd_dat1 | unused (tie high) | 85 |
| sd_dat2 | unused (tie high) | 80 |

## BSRAM Budget

The GW2AR-18 has 46 BSRAM blocks (2 KB each = 92 KB total). With ROM in BSRAM:

| Module | Size | Blocks |
|--------|------|--------|
| VRAM | 8 KB | 8 |
| WRAM | 8 KB | 4 |
| ExtRAM | 32 KB | 16 |
| ROM (new) | 32 KB | 16 |
| **Total** | **80 KB** | **44** |

Two blocks to spare. Tight but feasible.

## Module 1: SPI Byte Transport (`rtl/cart/sd_spi.sv`)

The SPI controller handles byte-level communication with the SD card. It
follows the same SPI Mode 0 (CPOL=0, CPHA=0) as our ST7789 LCD driver — MOSI
shifts on the falling edge, MISO is sampled on the rising edge.

Key features:
- **Two clock speeds**: ÷64 (~421 KHz) for initialization, ÷4 (~6.75 MHz)
  for data transfers. SD cards require slow clocking during the init handshake.
- **Full-duplex**: MOSI shifts out while MISO shifts in simultaneously.
- **External CS control**: The higher-level FSM controls chip select via
  `cs_en`, since CS must stay asserted across multi-byte command sequences.

```systemverilog
module sd_spi (
    input  logic       clk,
    input  logic       reset,
    output logic       sclk,
    output logic       mosi,
    input  logic       miso,
    output logic       cs_n,
    input  logic [7:0] tx_data,
    input  logic       start,
    output logic [7:0] rx_data,
    output logic       busy,
    output logic       done,
    input  logic       cs_en,
    input  logic       slow_clk
);
```

The clock divider counts to `clk_max` (63 for slow, 3 for fast), toggling
SCLK at the halfway point. A bit counter tracks the 8-bit transfer and pulses
`done` when complete.

One important detail: MISO is sampled directly (no pipeline register). Since
SCLK is generated by the FPGA, MISO transitions are synchronous to our clock
domain — an extra register would add unnecessary latency at the fast ÷4 speed.

## Module 2: SD Card Reader (`rtl/cart/sd_reader.sv`)

The reader handles the SD card initialization protocol and single-sector (512
byte) reads. It drives sd_spi to send/receive bytes.

### Initialization Sequence

SD cards require a specific handshake before data transfers:

1. **Power-up**: CS high, clock 80+ cycles (10× 0xFF bytes) at slow speed
2. **CMD0** (GO_IDLE_STATE): Assert CS, send command → expect R1=0x01 (idle)
3. **CMD8** (SEND_IF_COND): Voltage check with 0x1AA pattern → expect echo
4. **ACMD41 loop**: CMD55 + ACMD41(HCS=1) until R1=0x00 (card ready). This
   can take hundreds of milliseconds.
5. **CMD16** (SET_BLOCKLEN=512): Explicit block size for safety

### Sector Read (CMD17)

Once initialized, reading a sector is straightforward:

1. Assert CS, send CMD17 + sector number (SDHC uses sector addressing)
2. Wait for R1=0x00 response
3. Poll for data start token (0xFE)
4. Clock in 512 data bytes, pulsing `read_valid` for each
5. Clock in 2 CRC bytes (discarded)
6. Deassert CS, pulse `read_done`

```systemverilog
module sd_reader (
    input  logic        clk,
    input  logic        reset,
    // sd_spi interface
    output logic [7:0]  spi_tx,
    output logic        spi_start,
    input  logic [7:0]  spi_rx,
    input  logic        spi_busy,
    input  logic        spi_done,
    output logic        spi_cs_en,
    output logic        spi_slow_clk,
    // Sector read interface
    input  logic [31:0] sector,
    input  logic        read_start,
    output logic [7:0]  read_data,
    output logic        read_valid,
    output logic        read_done,
    output logic        ready,
    output logic        err
);
```

## Module 3: FAT32 Boot Loader (`rtl/cart/sd_boot.sv`)

The boot loader orchestrates the entire boot process. It uses sd_reader to read
sectors and **stream-parses** them byte-by-byte as they arrive — no 512-byte
buffer needed. The byte counter within each sector tells the parser which field
is being received.

### FAT32 Parsing

The state machine walks through four filesystem structures:

1. **MBR** (sector 0): Extract the first partition's LBA start address from the
   partition table at offset 446. Verify the partition type is FAT32 (0x0B or
   0x0C).

2. **VBR** (Volume Boot Record at partition start): Capture:
   - `sectors_per_cluster` (offset 13)
   - `reserved_sectors` (offset 14–15)
   - `num_fats` (offset 16)
   - `fat_size_32` (offset 36–39)
   - `root_cluster` (offset 44–47)
   - Compute: `fat_start = part_lba + reserved_sectors`
   - Compute: `data_start = fat_start + num_fats × fat_size_32`

3. **Root directory**: Read sectors for the root cluster, scan 32-byte
   directory entries. Skip deleted entries (0xE5), long filenames (attr=0x0F),
   directories, and volume labels. Match extension bytes 8–10 against "GB "
   (case-insensitive). Extract start cluster and file size.

4. **ROM data**: Follow the FAT cluster chain. For each cluster, read
   `sectors_per_cluster` sectors and write each byte to the BSRAM ROM via
   `rom_addr`/`rom_data`/`rom_wr`. After each cluster, read the FAT to find
   the next cluster in the chain. Stop at end-of-chain (≥ 0x0FFFFFF8) or after
   32768 bytes.

### Cluster-to-LBA Conversion

```systemverilog
function automatic [31:0] cluster_to_lba(input logic [31:0] cluster);
    return data_start + (cluster - 32'd2) * {24'd0, sectors_per_cluster};
endfunction
```

## gb_top Integration

### USE_SD Parameter

A new `USE_SD` parameter (default 0) controls whether SD boot is enabled:

- **USE_SD=0** (simulation): Distributed RAM ROM loaded via `$readmemh`.
  Boot completes immediately. Existing tests work unchanged.
- **USE_SD=1** (hardware): BSRAM ROM written by the SD boot loader. CPU
  held in reset until boot completes.

```systemverilog
module gb_top #(
    parameter int ROM_SIZE = 256,
    parameter     ROM_FILE = "sim/data/boot_test.hex",
    parameter int USE_SD   = 0
) ( ... );
```

### Reset Logic

The CPU reset now includes the boot state:

```systemverilog
wire hw_reset = !por_cnt[4];           // power-on + button reset
wire reset = hw_reset || !boot_done;   // also held during SD boot
```

### ROM Address Mux

When USE_SD=1, the ROM uses synchronous BSRAM with a write port for the boot
loader and a read port for the CPU:

```systemverilog
generate
    if (USE_SD != 0) begin : gen_rom
        wire [14:0] rom_bsram_addr = boot_done ? mbc_rom_addr[14:0] : sd_rom_addr;
        wire        rom_bsram_we   = !boot_done && sd_rom_wr;

        single_port_ram #(.ADDR_WIDTH(15), .DATA_WIDTH(8)) u_rom (
            .clk(clk), .we(rom_bsram_we),
            .addr(rom_bsram_addr), .wdata(sd_rom_data), .rdata(rom_rdata)
        );
    end else begin : gen_rom
        // Distributed RAM for simulation
        logic [7:0] rom_mem [0:ROM_SIZE-1];
        initial if (ROM_FILE != "") $readmemh(ROM_FILE, rom_mem);
        assign rom_rdata = rom_mem[mbc_rom_addr[$clog2(ROM_SIZE)-1:0]];
    end
endgenerate;
```

### BSRAM Wait States

Since the ROM is now synchronous BSRAM when USE_SD=1, it needs the same
1-cycle wait state as VRAM, WRAM, and ExtRAM:

```systemverilog
wire bsram_rd = (vram_cs || wram_cs || extram_cs || (USE_SD != 0 && rom_cs)) && cpu_rd;
```

## SD Card Simulation Model

Testing the SD card stack requires a behavioral model (`sim/model/sd_card_model.sv`)
that emulates an SD card's SPI responses:

- Detects 6-byte commands by watching MOSI with CS asserted
- Responds to CMD0 (R1=0x01), CMD8 (R7 with 0xAA echo), CMD55/ACMD41 (ready
  after a few attempts), CMD16 (R1=0x00)
- Returns 0xFE start token + 512 data bytes + 2 CRC for CMD17 reads
- Sector memory is pre-loaded by the test wrapper with FAT32 filesystem data

The test wrappers (`sim/top/sd_*_top.sv`) pre-load the model's sector memory
with a minimal FAT32 image: MBR, VBR, FAT table, root directory with a
"TETRIS  GB " entry, and ROM data sectors.
