# Tutorial 12 — ST7789 LCD Driver

Until now every test result has been a register value or a lit LED. To actually
see Game Boy graphics we need a display. The Tang Nano 20K doesn't have a
built-in SPI LCD like the smaller Tang Nano 9K, so we connect an external
ST7789 240×240 SPI module to the GPIO header with jumper wires.

In this tutorial we build the SPI controller, write the initialization
sequence, and stream a color-bar test pattern to the display — all generated
on-the-fly with no framebuffer RAM. The PPU and framebuffer come later.

**Source:** [ST7789V Datasheet](https://newhavendisplay.com/content/datasheets/ST7789V.pdf)

## Hardware Setup

The ST7789 module connects to the Tang Nano 20K's GPIO header with six wires:

| Signal    | Pin | ST7789 Module | Direction |
|-----------|-----|---------------|-----------|
| lcd_rst   | 71  | RST           | FPGA → LCD |
| lcd_cs    | 72  | CS            | FPGA → LCD |
| lcd_dc    | 73  | DC (RS)       | FPGA → LCD |
| lcd_sclk  | 74  | SCK (CLK)     | FPGA → LCD |
| lcd_mosi  | 75  | SDA (MOSI)    | FPGA → LCD |
| lcd_bl    | 76  | BLK (BL)      | FPGA → LCD |

Also connect VCC to 3.3V and GND to GND on the header. These pin numbers are
defaults — update `constraints.cst` if you wire to different pins.

## SPI Protocol

The ST7789 uses a 4-wire SPI interface:

- **Mode 3** (CPOL=1, CPHA=1): clock idles HIGH, data sampled on rising edge
- **MSB first**: bit 7 sent first
- **DC pin** selects between commands (DC=0) and data (DC=1)
- **CS** is active low

We divide the 27 MHz system clock by 4 to get a ~6.75 MHz SPI clock, well
within the ST7789's 62.5 MHz limit and reliable over jumper wires.

### SPI Byte Shifter

The shifter uses a 4-bit counter (`bit_cnt`) and an 8-bit shift register. On
each SPI falling edge it drives MOSI from the MSB; on each rising edge the
display samples the bit and the register shifts left:

```systemverilog
if (spi_falling) begin
    lcd_sclk <= 1'b0;
    lcd_mosi <= shift_reg[7];
end else if (spi_rising) begin
    lcd_sclk <= 1'b1;
    shift_reg <= {shift_reg[6:0], 1'b0};
    // advance or complete
end
```

Each byte takes 32 system clocks (8 bits × 4 clocks per bit).

## Initialization Sequence

After power-on, the ST7789 needs a specific command sequence:

| Step | Command | Hex  | Parameters | Wait After |
|------|---------|------|------------|------------|
| 1    | HW Reset | —  | RST low 10ms, high | 120ms |
| 2    | SLPOUT  | 0x11 | —         | 120ms |
| 3    | COLMOD  | 0x3A | 0x55 (RGB565) | — |
| 4    | MADCTL  | 0x36 | 0x00 (no rotation) | — |
| 5    | INVON   | 0x21 | — | — |
| 6    | CASET   | 0x2A | 0x00 0x28 0x00 0xC7 | — |
| 7    | RASET   | 0x2B | 0x00 0x30 0x00 0xBF | — |
| 8    | DISPON  | 0x29 | — | 120ms |
| 9    | RAMWR   | 0x2C | pixel data follows | — |

**INVON** is required because most ST7789 IPS panels have inverted color logic —
without it, colors appear as their complements.

**CASET/RASET** set the write window. We center the 160×144 Game Boy display on
the 240×240 panel: x offset = 40, y offset = 48, giving columns 40–199 and
rows 48–191.

### Init Command Table

Rather than a large case statement, the commands are stored in a ROM array with
a simple encoding:

```systemverilog
// Format: {type[1:0], data[7:0]}
//   00 = command byte (DC=0)
//   01 = data byte    (DC=1)
//   10 = delay (120ms)
//   11 = end marker
localparam int INIT_LEN = 22;
logic [9:0] init_rom [0:INIT_LEN-1];
```

The FSM walks through the table, sending command/data bytes via SPI and
inserting delays where marked.

## State Machine

The controller has a single top-level FSM:

```
RESET_LO  →  RESET_HI  →  INIT  ⇄  INIT_WAIT  →  DELAY
                                                       ↓
                         STREAM_HI  ←  (frame start from INIT end marker)
                            ↓
                         STREAM_LO
                            ↓
                        STREAM_WAIT  →  (next pixel or new frame)
```

- **RESET_LO/HI**: hardware reset with timing delays
- **INIT/INIT_WAIT**: walk the command table, send SPI bytes
- **DELAY**: timed 120ms pause (used after SLPOUT and DISPON)
- **STREAM_HI/LO/WAIT**: send each pixel as two RGB565 bytes

After the last pixel (159, 143), the FSM loops back to re-send the RAMWR
command and begins the next frame.

## Module Interface

```systemverilog
module st7789 (
    input  logic        clk,
    input  logic        reset,
    output logic        lcd_rst, lcd_cs, lcd_dc, lcd_sclk, lcd_mosi, lcd_bl,
    input  logic [15:0] pixel_data,  // RGB565 from external source
    output logic [7:0]  pixel_x,     // 0–159
    output logic [7:0]  pixel_y,     // 0–143
    output logic        pixel_req,   // strobed when pixel_data is sampled
    output logic        busy         // high during init
);
```

The pixel interface lets any source drive the display: a test pattern, a
framebuffer, or eventually the PPU. The controller outputs the current (x, y)
coordinates and strobes `pixel_req` when it captures the pixel.

## Top-Level Integration

In `gb_top.sv`, the LCD is instantiated alongside a simple test pattern:

```systemverilog
// Color bars: 8 vertical stripes
always_comb begin
    unique case (lcd_pixel_x[7:5])
        3'd0: lcd_pixel_data = 16'hF800; // red
        3'd1: lcd_pixel_data = 16'h07E0; // green
        3'd2: lcd_pixel_data = 16'h001F; // blue
        3'd3: lcd_pixel_data = 16'hFFE0; // yellow
        3'd4: lcd_pixel_data = 16'hF81F; // magenta
        3'd5: lcd_pixel_data = 16'h07FF; // cyan
        3'd6: lcd_pixel_data = 16'hFFFF; // white
        3'd7: lcd_pixel_data = 16'h0000; // black
    endcase
end

st7789 u_lcd (
    .clk(clk), .reset(reset),
    .lcd_rst(lcd_rst), .lcd_cs(lcd_cs), .lcd_dc(lcd_dc),
    .lcd_sclk(lcd_sclk), .lcd_mosi(lcd_mosi), .lcd_bl(lcd_bl),
    .pixel_data(lcd_pixel_data),
    .pixel_x(lcd_pixel_x), .pixel_y(lcd_pixel_y),
    .pixel_req(), .busy()
);
```

Change the pin numbers to match your wiring.

## Testing

The simulation testbench (`sim/test/st7789.zig`) uses the standalone
`st7789_top.sv` wrapper (no CPU, just the LCD controller with a test pattern).

Five tests verify the driver:

1. **Reset sequence** — `lcd_rst` goes low then high with correct timing
2. **SCLK idles high** — Mode 3 CPOL=1 verified after reset
3. **Init completes** — `busy` drops to 0 after all delays, backlight turns on
4. **SPI byte MSB first** — captures the first SLPOUT byte (0x11) bit by bit
5. **Pixel coordinates advance** — `pixel_req` fires and (x, y) increments

## Building and Running

Run just the LCD tests:

```
mise run test:st7789
```

Run the full test suite (103 tests across 14 modules):

```
mise run test
```

Synthesize and flash to see color bars on the display:

```
mise run build -- gb_top
mise run flash
```

## Performance

At 6.75 MHz SPI with 160×144 pixels (46,080 bytes per frame), each frame
takes about 55ms for a refresh rate of ~18 fps. This is plenty for a static
test pattern. Later tutorials can increase the SPI divider to ÷2 (13.5 MHz,
~37 fps) or use the full 27 MHz for ~60 fps when we need it.
